# secretNewsService 기획 명세서 v0.1

## 1. 문서 목적

본 문서는 secretNewsService의 1차 제품 범위와 업무 요구사항을 정의한다.

본 문서는 프로젝트명과 참고 문서(`whatdid_dev_final/docs`)의 업무 시스템 설계 패턴을 기반으로 작성한 뒤, 주노가 승인한 추천 기본값을 반영한 **v0.1 기준안**이다.

확정 기본값은 다음과 같다.

1. `secret`의 의미: 비공개/제한 공개 뉴스
2. v0.1 배포 채널: 웹 + 이메일
3. 인증 방식: 관리자/편집자/리뷰어는 Google OAuth, 독자는 이메일 OTP
4. AI 요약/분류/태그 추천: v0.1 제외
5. 외부 제보/외부 기고자: v0.1 제외

---

## 2. 제품 정의

secretNewsService는 제한된 사용자에게만 공개되는 뉴스/정보 콘텐츠를 수집, 작성, 검토, 승인, 공개, 배포, 보관하는 서비스다.

본 서비스는 단순 뉴스 게시판이 아니라 아래 기능을 포함하는 상태 기반 콘텐츠 운영 시스템이다.

- 뉴스 소스 수집 또는 수동 등록
- 내부 작성 및 편집
- 검토/승인 워크플로우
- 웹 조회 및 이메일 배포
- 비공개/제한 공개본 스냅샷 보존
- 배포 이력 저장
- 접근 권한 및 감사 로그 관리

---

## 3. 목표

1. 비공개 또는 제한 공개 뉴스 콘텐츠를 안전하게 관리한다.
2. 뉴스 작성부터 공개까지의 상태와 승인 흐름을 추적 가능하게 한다.
3. 공개 또는 배포된 콘텐츠는 이후 원본 수정과 독립된 스냅샷으로 보존한다.
4. 독자/고객/조직별 접근 범위를 서버에서 강제한다.
5. 뉴스 수집, 편집, 배포 과정의 모든 핵심 이벤트를 감사 로그로 남긴다.

---

## 4. 사용자 유형

### 4-1. 슈퍼 관리자
- 시스템 전체 설정을 관리한다.
- 관리자 계정과 권한을 관리한다.
- 예외적 삭제/복구 정책을 결정한다.

### 4-2. 시스템 관리자
- 사용자, 조직, 구독/초대, 뉴스 카테고리, 배포 채널을 관리한다.
- 전체 뉴스 상태를 조회하고 강제 보관 또는 공개 중단을 수행할 수 있다.

### 4-3. 편집자
- 뉴스 초안을 작성한다.
- 외부 소스에서 수집된 후보 뉴스를 선별한다.
- 뉴스 본문, 요약, 태그, 공개 범위를 수정한다.
- 검토 요청을 제출한다.

### 4-4. 리뷰어 / 승인자
- 편집자가 제출한 뉴스를 검토한다.
- 승인 또는 반려한다.
- 승인된 뉴스를 공개 예약 또는 즉시 공개할 수 있다.

### 4-5. 독자
- 본인에게 허용된 뉴스만 조회한다.
- 공개 범위가 맞지 않는 뉴스는 목록과 상세 모두 접근할 수 없다.

### 4-6. 외부 제보자 / 외부 기고자 — v0.1 제외
- v0.1에서는 외부 제보자/외부 기고자 역할을 제공하지 않는다.
- 외부 제보 또는 기고 워크플로우는 후속 버전 후보로 둔다.

---

## 5. 핵심 업무 범위

### 5-1. 뉴스 수집
- 관리자는 뉴스 소스를 등록할 수 있다.
- 시스템은 등록된 소스에서 뉴스 후보를 수집할 수 있다.
- 수집 실패, 중복 후보, 차단된 소스는 별도 상태로 기록한다.

### 5-2. 뉴스 작성
- 편집자는 수동으로 뉴스 초안을 생성할 수 있다.
- 수집된 후보를 초안으로 전환할 수 있다.
- 제목, 요약, 본문, 원문 링크, 태그, 민감도, 공개 범위, 배포 채널을 입력한다.

### 5-3. 검토/승인
- 편집자는 초안을 검토 요청 상태로 전환할 수 있다.
- 리뷰어는 승인 또는 반려할 수 있다.
- 반려 시 반려 사유를 필수 기록한다.
- 승인된 뉴스만 공개 가능하다.

### 5-4. 공개/배포
- 승인된 뉴스는 즉시 공개 또는 예약 공개할 수 있다.
- 공개 시점에 공개본 스냅샷을 생성한다.
- 웹 공개와 이메일 배포를 v0.1 필수 채널로 한다.
- Discord, Slack, Telegram 배포는 v0.1에서 제외한다.
- 이메일 배포 결과는 수신자 단위로 기록한다.
- 배포가 시작된 공개본은 본문 수정이 불가하다. 정정은 새 revision 또는 정정 공지로 처리한다.

### 5-5. 조회
- 독자는 본인 권한 범위 내 뉴스 목록과 상세를 조회한다.
- 관리자는 상태, 카테고리, 태그, 공개 범위, 작성자, 승인자 기준으로 검색한다.
- 공개본은 스냅샷 데이터를 기준으로 렌더링한다.

### 5-6. 댓글/피드백 — 선택 범위
- 내부 검토 댓글과 공개 후 독자 피드백은 분리한다.
- 내부 댓글은 외부 독자에게 노출하지 않는다.
- 공개 피드백은 별도 정책에 따라 허용/차단한다.

### 5-7. 감사 로그
아래 행위는 감사 로그를 남긴다.

- 로그인
- 뉴스 생성/수정/삭제 또는 보관
- 상태 전이
- 승인/반려
- 공개/공개 중단
- 배포 시도/성공/실패
- 권한 변경
- 민감 뉴스 조회

## 6. 뉴스 상태 초안

### 6-1. 콘텐츠 상태
- `DRAFT`: 작성 중
- `REVIEW_REQUESTED`: 검토 요청됨
- `REJECTED`: 반려됨
- `APPROVED`: 승인됨
- `SCHEDULED`: 공개 예약됨
- `PUBLISHED`: 공개됨
- `ARCHIVED`: 보관됨
- `WITHDRAWN`: 공개 중단됨

### 6-2. 수집 상태
- `CANDIDATE`: 수집 후보
- `IMPORTED`: 초안 전환됨
- `DUPLICATE`: 중복
- `IGNORED`: 제외
- `FETCH_FAILED`: 수집 실패

### 6-3. 배포 상태
- `NOT_SENT`: 미발송
- `SENDING`: 발송 중
- `SENT`: 전체 성공
- `PARTIAL_FAILED`: 일부 실패
- `FAILED`: 전체 실패

---

## 7. 권한 원칙

1. 서버가 최종 권한 판정을 수행한다.
2. 뉴스 접근 권한은 사용자 역할 + 조직/구독/초대/공개 범위 조합으로 판단한다.
3. 민감도 높은 뉴스는 관리자 또는 승인된 독자 그룹만 접근할 수 있다.
4. 공개 또는 배포가 시작된 뉴스 본문은 직접 수정할 수 없다.
5. 관리자도 권한 밖 조직의 독자 데이터를 임의 조회하지 못하게 범위를 제한할 수 있다.
